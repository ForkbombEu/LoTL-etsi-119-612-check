## Implementation Notes

- Stable findings are `ts119602.syntax.uri`, `ts119602.syntax.date_time`, `ts119602.syntax.language`, `ts119602.syntax.country_code`, and `ts119602.language.annex_b`.
- Date-time validation now requires `YYYY-MM-DDThh:mm:ssZ`, rejects fractional seconds and offsets, and validates calendar/clock values before ordering checks.
- URI validation checks absolute RFC 3986 lexical constraints plus local HTTP(S), `mailto`, and `tel` syntax. Registered Annex C values remain a separate requirement.
- The pinned country policy contains 249 ISO 3166-1 alpha-2 identifiers, requires `UK` and `EL` instead of `GB` and `GR`, accepts `EU`, and records the standard's `AP`, `ASIA`, `GCC`, and `ASEAN` grouping examples.
- Unlisted uppercase regional or multi-state identifiers produce `inconclusive` because clause 6.1.5 depends on external recognition evidence.
- Multilingual validation checks lower-case RFC 5646 syntax, an English `en` entry, duplicate tags, non-empty values, and locally observable Annex B character restrictions.

## Known Caveats

- The URI requirement remains marked partial because exact registered/profile URI comparison belongs to Annex C and later profile tasks.
- Language and Annex B requirements remain partial: native-term transliteration, source-byte encoding, combining-character recommendations, dereferenced pointer content, and parser interoperability are not checked.
- Country validation cannot establish arbitrary regional or multi-state recognition without additional policy evidence.
- This task does not implement address semantics, XML XSD validation, signature validation, or Annex D-I profile dispatch.
