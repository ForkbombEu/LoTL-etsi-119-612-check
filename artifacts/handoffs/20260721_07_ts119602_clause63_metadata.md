## Implementation Notes

- Added a binding-neutral metadata observation model and stable findings shared by JSON and XML assessment paths.
- Implemented the clause 6.3/Table 1 implicit/explicit presence matrix, local integer checks, `CC:name`, postal/email/website address structure, policy/legal-notice alternatives, history-period semantics, pointer shape, date ordering/expiry/closure, distribution URI structure, and scheme-extension criticality.
- Contextual sequence progression, pointer authentication, profile maximum update intervals, and distribution byte equality remain explicitly `not_checked`.
- Added a versioned standards-interpretation registry for known V1.1.1 document/schema conflicts. Closed-list `NextUpdate` semantics cite the applicable registry entry while retaining the independent JSON schema diagnostic.
- Added a versioned empty scheme-extension registry; unknown critical scheme extensions fail closed, while unknown non-critical extensions may be ignored.
- Removed the generic duplicate-language rejection because a multilingual contact collection can contain distinct email and website URI entries with the same language.

## Known Caveats

- Mode inference is local and based on explicit-only Table 1 fields; profile selection and Annex C exact URI values are not implemented.
- The scheme-extension registry currently recognizes no extensions, so every critical scheme extension is rejected until an explicit supported extension is registered.
- Pointer target authentication and distribution-point dereferencing require bounded external evidence and are deferred to TS602-12.
- The six-calendar-month rule is profile-specific and remains deferred to Annex D-I profile dispatch.
- Entity/service status and history consequences, plus entity/service critical extensions, remain TS602-08 work.
